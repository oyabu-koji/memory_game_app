# アーキテクチャ設計書 (Architecture Design Document)

## 前提と方針

このリポジトリは現在、Vitest / ESLint を備えたテンプレート状態にある。既存の TypeScript 関連ファイルは移行時の整理対象とし、アプリ実装言語は JavaScript を採用する。  
実装フェーズでは、既存の品質基盤を活かしつつ、モバイルアプリ本体を React Native + Expo + JavaScript で構築する。  
MVPでは「単一のオフラインクライアントアプリ」として設計し、サーバーや外部サービスへの依存を持たない。

### フレームワーク・ライブラリ

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| React Native | Expo管理の対応版 | モバイルUI実装 | iOS / Android向けUIを単一コードベースで実装できる |
| Expo | 導入時点の安定版を固定 | アプリ実行基盤、端末機能アクセス | ネイティブ設定の負担を抑えつつ、音や振動を扱いやすい |
| React | Expo同梱版 | 宣言的UI | 状態変化と画面表現を分離しやすい |
| React Native Animated API | React Native同梱 | カード反転、クリア演出 | MVPの演出には十分で、依存追加を最小化できる |
| Expo管理の音 / 振動モジュール | Expo同梱または公式対応版 | 効果音、触覚フィードバック | 端末差異を吸収しながらオフラインで動作させやすい |

## アーキテクチャパターン

### Feature-first + Layered Client Architecture

```
┌──────────────────────────────────────┐
│ UI Layer                             │
│ screens / components                 │
├──────────────────────────────────────┤
│ Application Layer                    │
│ hooks / session controller           │
├──────────────────────────────────────┤
│ Domain Layer                         │
│ game rules / deck generation         │
├──────────────────────────────────────┤
│ Platform Layer                       │
│ audio / haptics / bundled assets     │
└──────────────────────────────────────┘
```

#### UI Layer
- **責務**: 子供向け画面の描画、タップイベントの受け取り、進行状況の表示
- **許可される操作**: Application Layerのhookやcontrollerを呼ぶ
- **禁止される操作**: ドメインロジックを書き込む、端末APIを直接叩く

#### Application Layer
- **責務**: ゲーム状態遷移の調停、UIイベントとドメインロジックの接続、非同期演出の順序管理
- **許可される操作**: Domain LayerとPlatform Layerを呼ぶ
- **禁止される操作**: 画面描画の詳細を持つこと

#### Domain Layer
- **責務**: デッキ生成、カード選択制約、マッチ判定、ゲーム終了判定
- **許可される操作**: 純粋関数・JSDocで表現した構造定義のみに依存する
- **禁止される操作**: React hook、端末API、画面依存

#### Platform Layer
- **責務**: 音、振動、画像アセット、将来的なローカル保存など端末依存処理の抽象化
- **許可される操作**: Expo / React Nativeの端末機能を利用する
- **禁止される操作**: ゲームルールの意思決定を持つこと

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| ゲーム進行状態 | メモリ上の状態 | JavaScriptオブジェクト | MVPでは1セッション完結で十分、保存処理を持たずシンプルに保てる |
| カード画像・効果音 | アプリ内バンドルアセット | PNG / WebP / MP3等 | オフライン必須のため、起動時点で端末内に存在する必要がある |
| 将来の設定値 | 端末内ローカル保存を予定 | JSON相当 | 音量や難易度を追加する場合の拡張先として確保する |

### バックアップ戦略

- MVPではユーザー生成データを永続化しないため、バックアップや復元は不要
- 将来的に設定や成績保存を追加する場合のみ、ローカル保存の破損耐性と初期値復元を設計する

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| コールドスタートから最初の操作可能状態 | 2秒以内 | 中価格帯スマートフォン実機 |
| カードタップから反転開始 | 100ms以内 | 実機、リリース相当ビルド |
| 2枚目選択後の一致判定反映 | 100ms以内 | 実機、リリース相当ビルド |
| ミスマッチ演出を含むターン完了 | 1.2秒以内 | 実機、リリース相当ビルド |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 150MB以内を目安 | 子供向け端末でも安定して動作させるため |
| CPU | プレイ中に継続的な高負荷状態を作らない | アニメーション中の発熱やカクつきを避けるため |
| アセット容量 | MVPで20MB以内を目安 | 配布サイズとロード時間を抑えるため |

## セキュリティアーキテクチャ

### データ保護

- **個人情報**: MVPでは扱わない
- **外部通信**: 実装しない。ゲーム進行にネットワークを必要としない
- **権限要求**: 音・振動に必要な最小限にとどめ、位置情報・カメラ・マイクは要求しない

### 入力検証

- タップ対象は「裏向きかつ未マッチのカード」のみに限定する
- `selectedCardIds` の件数は常に2件以下を保つ
- 無効な状態遷移が発生した場合は、状態を破壊せず現在状態を維持する

### エラーハンドリング

- 音や振動の利用失敗は recoverable error として扱い、ゲームは続行する
- アセット不足や不整合は開発時に検出し、ユーザー向けにはクラッシュよりもフォールバック表示を優先する

## スケーラビリティ設計

### データ増加・機能増加への対応

- 盤面サイズは `GameConfig` (`rows`, `columns`, `pairCount`, `mismatchDelayMs`) で変更可能な設計にし、4 x 5以外の難易度追加に備える
- カード絵柄セットはアセットカタログ化し、テーマ追加時にロジック変更を最小化する
- 成績保存や設定保存を追加する場合は Platform Layer に閉じ込める

### 機能拡張性

- **テーマ拡張**: 絵柄と演出セットを切り替え可能にする
- **難易度拡張**: ペア数、盤面サイズ、演出量を設定で切り替えられるようにする
- **ローカライズ**: テキストを最小限の辞書ファイルに集約し、多言語化しやすくする

## テスト戦略

### ユニットテスト
- **フレームワーク**: Vitest
- **対象**: `createDeck`、`resolveTurn`、ゲーム終了判定、入力ガード
- **カバレッジ目標**: Domain Layer 90%以上

### コンポーネントテスト
- **フレームワーク**: Vitest + `@testing-library/react-native`
- **対象**: `HomeScreen`、`GameScreen`、`GameBoard` と `useGameSession` をまたぐ主要フロー
- **確認内容**: ゲーム開始、2枚目選択後の入力ロック、マッチ成立、ミスマッチ復帰、クリア導線
- **制約**: 音、振動、オフライン起動のようなネイティブ依存挙動はこの層では保証しない

### 実機検証
- **対象**: 音、振動、アニメーション、オフライン起動
- **方針**: iOS / Androidの少なくとも各1端末で、主要ユーザーフローを手動確認する
- **記録先**: `.steering/[YYYYMMDD]-[task]/tasklist.md`

## 技術的制約

### 環境要件
- **対象OS**: iOS / Android
- **画面向き**: MVPは縦向き前提
- **必要な外部依存**: なし。ネットワーク不要
- **開発環境**: Node.js 22.x、npm 10.x、JavaScript (ES2022)

### パフォーマンス制約
- 端末性能に依存しないよう、同時アニメーション数を必要最小限に絞る
- 音声は短い効果音中心とし、長尺BGMを前提にしない

### セキュリティ / 品質制約
- トラッキングSDKや広告SDKは導入しない
- ゲームロジックと端末APIを分離し、テストと将来差し替えを容易にする

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| Expo / React Native関連 | モバイル実行基盤 | Expo互換性を優先し、対応組み合わせを固定する |
| JavaScript (ES2022) | アプリ実装言語 | Expo / Node の標準サポート範囲に寄せ、追加の変換設定を増やしすぎない |
| Vitest | テスト | 現行メジャーバージョン内で管理する |
| `@testing-library/react-native` | コンポーネントテスト | React Native / runner 互換性を優先して採用する |
| 追加ライブラリ全般 | 補助機能 | 標準機能で代替できない場合のみ導入し、理由を `.steering/` に残す |
